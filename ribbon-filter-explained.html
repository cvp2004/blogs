00 · Introduction

If you've never heard of a Ribbon filter

This is a from-scratch introduction. You need no background in databases, no maths beyond counting, and no idea what any of the words in the title mean yet. We'll build every piece together, in order.

Start with the everyday problem it solves. Imagine you keep a huge collection of things — names, IDs, keys, whatever — and you constantly need to answer one yes-or-no question: “is this particular thing in my collection?” Searching the real collection every time is slow. A Ribbon filter is a tiny stand-in that lives in memory and answers that question almost instantly, while taking up a remarkably small amount of space.

The catch — and the charm — is that it's allowed to be wrong, but only in one harmless direction. It will never miss something that's truly in the collection. Just occasionally it raises a false alarm: it says “might be here” about something that isn't, which costs you nothing but a quick double-check.

If you want a picture for it: think of a club bouncer holding a guest list. They can instantly turn away anyone whose name is definitely not on the list, and only rarely wave through a gate-crasher whose name happens to resemble a real guest's. Fast, compact, and wrong only in the safe direction. That's the whole spirit of the thing.

Why should you care? Because these little gadgets sit quietly underneath most modern data systems — databases like RocksDB, columnar file formats like Parquet — saving disk reads and memory at enormous scale. The Ribbon filter is a recent and especially space-thrifty member of that family, and once you see how it works, it's genuinely elegant.

where this is going We'll start with three tiny building blocks — bits, hashing, and XOR — none of which is hard. Then we'll assemble them into the older Bloom filter, see where it's wasteful, and finally reshape it into the Ribbon filter. By the end, the page's title will read as plain English.

01 · The problem

Why would anyone need this?

Imagine a database that has spread your data across hundreds of files sitting on a disk. You ask it a simple question: “Is the key user:42 stored in this particular file?”

The obvious way to find out is to open the file and look. But opening a file on disk is slow — thousands of times slower than reading something already in memory. If the key turns out not to be there, you've paid that cost for nothing. Now multiply that by hundreds of files and millions of lookups.

What we really want is a tiny helper that lives in memory and that we can ask first, before touching the disk. It should answer one of two things:

the deal it offers “Definitely not here” — don't bother opening the file, or
“Maybe here” — go ahead and look, it might be.

That helper is called an approximate membership filter. The Ribbon filter is one particular, very space-efficient design of such a helper. Everything below builds toward understanding exactly how it works — and we'll assume you know none of the pieces yet.

02 · The rules of the game

False alarms are fine. Missed keys are not.

Notice the lopsided promise above. The filter never says a confident “yes” — only “definitely not” or “maybe.” That asymmetry is the whole point, so let's name the two kinds of mistakes a filter could make.

A false negative is when a key really is in the set but the filter says “not here.” This would be a disaster: you'd skip a file that actually holds your data, and the database would return wrong answers. So we design the filter to make this impossible.

A false positive is when a key is not in the set but the filter says “maybe.” This is merely annoying: you open a file, find nothing, and shrug. You wasted one disk read. We can't eliminate these entirely, but we can make them rare.

the entire engineering goal Push the false-positive rate as low as possible while spending as few bits of memory as possible. Every filter — Bloom, XOR, Ribbon — is just a different answer to that one trade-off.

Before we can see how Ribbon achieves it, we need three small building blocks. None of them is hard.

03 · Building block one

Hashing: turning anything into numbers

A hash function is a machine that takes any input — a word, a key, a whole file — and spits out a number. It has two properties we care about:

It's deterministic. The same input always produces the same output. Feed in user:42 today and tomorrow, you get the identical number both times. This matters because we'll hash a key once when storing it and again when looking it up, and we need those to agree.

It scrambles. Inputs that look similar (user:42 and user:43) produce wildly different, unrelated-looking outputs. The numbers behave as if they were random — even though they're perfectly repeatable.

That's all a hash is for our purposes: a repeatable way to turn a key into some numbers that behave randomly. We'll lean on it constantly.

04 · Building block two

XOR: addition that forgets to carry

Computers store everything as bits — switches that are either 0 or 1. The Ribbon filter leans on one operation between bits called XOR, written with the symbol ⊕. The rule is simple: the result is 1 only when the two inputs differ.

aba ⊕ b
000
011
101
110

You can think of XOR as adding without carrying: 1 + 1 would normally carry to make 2, but XOR just wraps back to 0. To XOR two multi-bit numbers, you XOR each column independently. Three properties make XOR magical for our purposes:

Order doesn't matter. a ⊕ b equals b ⊕ a, and you can XOR a whole list in any order.

It's its own undo. XOR something twice and it cancels: a ⊕ a = 0. This is the secret behind everything that follows.

It's reversible. If a ⊕ b = c, then knowing any two of them gives you the third. That “solve for the missing one” ability is exactly what lets us fill in unknowns later.

05 · The old champion

The Bloom filter, and why it's a little wasteful

Before Ribbon, the standard answer was the Bloom filter. Picture a long row of light switches, all off. To insert a key, you hash it a few times to pick (say) three switches, and flip them all on. To look up a key, you hash it the same way and check its three switches: if any one is off, the key was definitely never inserted; if all three are on, it's “probably” there.

Bloom filters work well and are wonderfully simple. But they pay a tax: to hit a given false-positive rate, they use about 44% more memory than the theoretical minimum. The waste comes from switches being shared sloppily — many keys flipping overlapping switches, with no coordination. Ribbon filters squeeze almost all of that waste out. To see how, we need a completely different mental model than light switches.

06 · The core idea

What if each key were a little equation?

Throw away the light switches. Instead, picture a long row of lockers, each holding a small number written on a slip of paper.

Now we hand every member of our set a personal rule. We use the hash from block three to give each key two things: which lockers are “theirs” and a short secret fingerprint (just some bits derived from the key). The rule for a key X reads:

one key's rule “Take my lockers, XOR their numbers together, and the result must equal my fingerprint.”

Here's the shift in thinking. We have many keys, so we have many such equations — all sharing the same row of lockers. The job of building the filter is to fill in every locker's number so that all the equations are true at once. That's just solving a system of equations, the same thing you did in school with x and y — except the arithmetic is XOR instead of ordinary +.

And looking up a key becomes trivial: re-hash it to find its lockers and fingerprint, XOR the lockers, and check whether you get the fingerprint back. We'll see exactly why that gives no false negatives in a moment.

07 · A gentle look at the math

Solving a system, the XOR way

You don't need to love algebra for this — just one picture. Each equation says “a certain set of lockers XORs to a certain value.” If two equations share lockers, we can combine them by XOR-ing the equations together. Because a ⊕ a = 0, any locker that appears in both equations cancels out, leaving a simpler equation with fewer unknowns.

Repeat that cancelling process and you eventually whittle each equation down until it pins one locker to one definite value. Then you work backwards, filling in the rest. This is the XOR version of the “Gaussian elimination” you may have heard of — the standard recipe for solving systems of linear equations.

why bother with equations? Because the solution can be packed extremely tightly. A Bloom filter wastes ~44% over the theoretical minimum. This equations approach (the family includes “XOR filters”) gets that down to about 23% — and Ribbon, as we'll see, pushes it to ~10% or less.

So why isn't this the whole story? Because there's a practical catch. If each key's lockers are scattered randomly across the entire row, then solving the system means jumping all over memory — terrible for the computer's cache — and the bookkeeping gets expensive. The Ribbon filter's one brilliant idea is a fix for exactly this.

08 · The Ribbon trick

Banding: keep every equation local

The name spells out the trick: Rapid Incremental Boolean Banding ON the fly. The word that matters is banding.

Instead of letting a key's lockers scatter everywhere, we confine them to a short contiguous window — say four lockers in a row — starting wherever the key's hash lands. A key's equation only ever touches lockers inside its own little window.

Watch what happens when you line up several keys, each with its window starting a bit further along. Press the button to send them from scattered (the old way) into their bands (the Ribbon way):

That single constraint — keep each equation inside a narrow diagonal band — buys two enormous wins:

It's cache-friendly. Every operation touches a small run of neighbouring lockers, not the whole row. The computer loves this.

It makes the puzzle solvable in basically one pass. Because the equations form a tidy band rather than a tangled mess, the Gaussian elimination runs in nearly straight-line time instead of the slow, all-over-the-place version. Next we'll walk through exactly how that one-pass build works.

09 · Construction, line by line

Building the filter, one key at a time

“Incremental, on the fly” means we feed in members one by one. For each, we look at its band and try to give it a home. Here's the heart of it in plain pseudocode. Read each line with its note on the right — the logic is small.

build( members ) → solved lockers
# Phase 1 — collect equations into a banded matrix
1
for each key in members:
Process members one at a time, in any order.
2
start, band, fp = hash(key)
The hash gives a starting locker, the pattern of which lockers in its window the equation touches (the “band”), and the fingerprint the window must XOR to.
3
pos = start
We'll try to “anchor” this equation at the first locker it touches.
4
while band ≠ 0:
Keep going until the equation has found a home or vanished.
5
if pivot[pos] is empty:
Is any earlier equation already “anchored” at this locker? If not, this slot is free.
6
pivot[pos] = (band, fp)
Claim it. Store this equation here and move on to the next key. Done.
7
break
 
8
else:
The slot is taken. We must untangle the collision.
9
band ^= pivot[pos].band
XOR our equation with the one already living here. Because a⊕a=0, the shared leading locker cancels.
10
fp ^= pivot[pos].fp
Keep the right-hand side consistent — XOR the fingerprints too.
11
advance pos to next set bit
The cancellation slides our equation's leading edge rightward — still inside the narrow window. Loop and try the new slot.
12
if band == 0 and fp ≠ 0:
Everything cancelled but the answer isn't zero — a contradiction. This batch of keys has no solution.
13
restart with a new hash seed
Rare. Just reshuffle which lockers keys map to and rebuild. Cheap to detect, cheap to retry.
# Phase 2 — back-fill the actual locker values
14
for pos from right to left:
Walk backwards through the anchored equations.
15
locker[pos] = pivot[pos].fp ⊕ (already-known lockers in band)
Each equation now pins exactly one locker, because everything to its right is already filled. Solve for it. This is the “reversible” property of XOR doing the work.

The key insight in lines 5–11: because every band is only a few lockers wide, a collision can only push an equation a handful of steps to the right before it settles. There's no wandering across the whole row. That's why the whole build is nearly linear in the number of keys.

10 · Querying, line by line

Looking a key up

Querying is the easy direction. We never stored the keys, so we simply recompute the equation and check it.

contains( key ) → "maybe" or "definitely not"
1
start, band, fp = hash(key)
Exactly the same hash as during the build — so a real member regenerates the very window and fingerprint it was stored with.
2
result = xor_of(lockers in this key's band)
XOR together the handful of lockers the key's window covers.
3
if result == fp:
Does the window XOR to the fingerprint?
4
return "maybe present"
For a real member this is guaranteed — we filled the lockers precisely to make it so. Hence: no false negatives, ever.
5
else:
 
6
return "definitely not present"
A member could never land here. So a mismatch is a rock-solid “no.”

Now let's watch the whole lifecycle end to end. First the four members get inserted — watch the lockers fill in, and watch each new key adjust exactly one locker so its own rule comes true without disturbing the keys before it. Then we'll look keys up, including one non-member that slips through as a false positive.

That's the whole machine. user:42 and post:7 are real members and always match. user:13 is correctly turned away. And ghost:5 — never inserted — slips through because its window just happens to XOR to its fingerprint by pure coincidence. That's a false positive: harmless (one wasted disk read), but real. How often it happens is the last dial to understand.

11 · Tuning the mistakes

Longer fingerprints, fewer false alarms

For a key that isn't a member, the XOR of its window is essentially a random number. It collides with the key's fingerprint only by luck. If the fingerprint is b bits long, the chance of an accidental match is one in 2b.

So the false-positive rate is a dial you set directly: every extra bit you spend on fingerprints halves the false-positive rate. Drag the slider to feel it:

Our toy demo used a tiny 3-bit fingerprint so the XOR was readable by eye — which is exactly why ghost:5 collided so easily. Real implementations use longer fingerprints, where coincidences are vanishingly rare.

12 · The payoff

How much memory does this save?

This is the reason Ribbon exists. The bars show how much memory each design uses above the theoretical minimum needed to hit a given false-positive rate. Shorter is better — closer to the unbeatable lower bound.

For a database holding billions of keys, the gap between Bloom's +44% and Ribbon's ~+10% is a lot of RAM saved — RAM that can hold more useful data instead.

13 · The catch & the real world

Nothing is free

Ribbon's space savings come at a cost: building it takes more CPU than a Bloom filter. You're solving a system of equations, not just flipping switches. Queries are a touch heavier too.

The band width is the master dial. Wider bands pack the lockers tighter (less wasted space) but cost more CPU per insert. Narrower bands build faster but waste slightly more space. You tune it to your situation.

That trade-off explains where Ribbon is actually used. It originated in RocksDB (a widely used storage engine) around 2021. The guidance there: use Ribbon filters for the lower, larger, long-lived levels of the data structure — the data that sits on disk a long time, where the space savings are huge and you only build the filter once, so the extra CPU is amortised away. For the top levels that get rewritten constantly, a plain Bloom filter often still wins, because build speed matters more than space there.

if you've heard of LSM-trees This is the classic Ribbon-vs-Bloom decision: Bloom for the hot, frequently-rewritten upper levels; Ribbon for the cold, stable, space-dominated lower levels. Same filter job, different point on the space–CPU trade-off.
14 · The one sentence

Putting it all back together

You now have every piece. A Ribbon filter:

① gives each key one XOR equation — “my lockers must XOR to my fingerprint” — using a hash to pick the lockers and the fingerprint. ② confines each equation to a short window, so all the equations stack into a tidy diagonal band. ③ solves that banded system in nearly one pass to fill the lockers. ④ answers a lookup by re-XOR-ing a key's window and comparing to its fingerprint — always right for members, occasionally a harmless false alarm for non-members, at a rate you set with the fingerprint length.

the whole thing, compressed A Ribbon filter answers “is this key in the set?” by storing the solution to a big XOR puzzle in which each key is one equation confined to a small sliding window — and that “windowed equations” structure is what makes it both fast and unusually compact.

That's it. The same diagonal stripe you saw at the very top is the whole idea — every locker satisfying every overlapping equation at once.
